# Notifications

This guide is for module authors who want their feature to send notifications.
It covers the data model, the two delivery paths (recurring + immediate), and
the wiring you need to add when introducing notifications in a new module.

For VAPID key generation, service-worker setup, and ops/deployment concerns,
see [`NOTIFICATIONS_SETUP.md`](./NOTIFICATIONS_SETUP.md). That doc predates the
aepbase migration and parts are stale, but the VAPID + service-worker steps
still apply.

## Architecture

There are three aepbase collections, all parented under `user`
(see [`aepbase/terraform/notifications.tf`](../aepbase/terraform/notifications.tf)):

| Collection                  | URL pattern                                | Purpose                                                                 |
|-----------------------------|--------------------------------------------|-------------------------------------------------------------------------|
| `notifications`             | `/users/{uid}/notifications/{id}`          | One row per delivered notification (the user's inbox)                   |
| `notification-subscriptions`| `/users/{uid}/notification-subscriptions/{id}` | One row per browser/device push endpoint (created from Settings)    |
| `recurring-notifications`   | `/users/{uid}/recurring-notifications/{id}`| Template that generates `notifications` on a schedule from a source row |

Because every collection is user-parented, regular users only ever see their
own rows. There's no `user_id` filter to write — the URL implies it. The
`user_id` field on `notification` rows is kept as a denormalized convenience.

The frontend exports collection names from `@/core/api/aepbase`:

```ts
AepCollections.NOTIFICATIONS              // 'notifications'
AepCollections.NOTIFICATION_SUBSCRIPTIONS // 'notification-subscriptions'
AepCollections.RECURRING_NOTIFICATIONS    // 'recurring-notifications'
```

## Data model

### `notification`

A delivered notification. Created by the server when a push goes out (or
attempted). Read by the UI to populate the bell / inbox.

Key fields (`frontend/src/modules/notifications/types.ts`):

- `title`, `message` — what the user sees
- `notification_type` — `day_of` | `day_before` | `week_before` | `system`
- `read`, `read_at` — read state, set client-side via
  `useMarkNotificationAsRead`
- `source_collection`, `source_id` — what record this is about
  (e.g. `people` / `<personId>`); used for icons and deep links
- `recurring_notification_id` — present when generated by a recurring template
- `person_id` — **deprecated**; prefer `source_collection`+`source_id`

### `recurring-notification`

A template that the daily cron evaluates against a source record.

```ts
{
  source_collection: 'people',           // any aepbase collection name
  source_id: '<recordId>',
  title_template: 'Birthday — {{name}}',
  message_template: "{{name}}'s birthday is on {{date}}",
  reference_date_field: 'birthday',      // field on the source record
  timing: 'day_of',                      // | 'day_before' | 'week_before'
  enabled: true,
  last_triggered: '2026-04-28T09:00:00Z',
}
```

Templates support `{{name}}` (resolved from `sourceRecord.name`) and
`{{date}}` (resolved by formatting `sourceRecord[reference_date_field]`).
See `processTemplate` in
`frontend/src/app/api/notifications/utils/notification-sender.ts`.

### `notification-subscription`

A web-push endpoint. Settings → Notifications creates these when the user
clicks "Enable Push Notifications". Module authors generally do not touch
this collection directly — read or send through the helpers below.

## Delivery path 1: recurring notifications

Use this when something in your module has a date that should fire a yearly
reminder (birthdays, renewals, service dates, expiry).

### How the cron works

1. Vercel Cron (or any scheduler) hits `GET /api/notifications/cron` once a
   day. The endpoint requires `Authorization: Bearer ${CRON_SECRET}` unless
   running locally.
2. The route logs in to aepbase as the admin user
   (`AEPBASE_ADMIN_EMAIL` / `AEPBASE_ADMIN_PASSWORD`), then iterates every
   user's enabled `recurring-notifications`.
3. For each row it loads the source record at
   `${source_collection}/${source_id}`, reads `reference_date_field`, and
   checks whether `timing` lines up with today (yearly recurring — month/day
   match for the current year).
4. If the timing matches, it pushes to every `enabled` subscription, writes
   a `notifications` row, and updates `last_triggered`. Expired subscriptions
   (HTTP 404/410) are pruned.

The route ([`frontend/src/app/api/notifications/cron/route.ts`](../frontend/src/app/api/notifications/cron/route.ts))
calls `checkAndSendRecurringNotifications` in
[`utils/notification-sender.ts`](../frontend/src/app/api/notifications/utils/notification-sender.ts).
A new module needs no server-side code as long as its records expose the
`reference_date_field` you point templates at.

### Creating templates from your module

You have two options. For most cases, use the generic mutation hooks from
`@/modules/notifications`:

```ts
import {
  useSyncRecurringNotifications,
  useRecurringNotifications,
} from '@/modules/notifications';
import type { NotificationTiming } from '@/modules/notifications';

const sync = useSyncRecurringNotifications();

await sync.mutateAsync({
  sourceCollection: 'gift_cards',
  sourceId: card.id,
  referenceDateField: 'expires_at',
  titleTemplate: 'Gift card expiring — {{name}}',
  messageTemplate: '{{name}} expires on {{date}}',
  desiredTimings: ['week_before', 'day_before'] as NotificationTiming[],
});
```

`useSyncRecurringNotifications` reconciles per
`(sourceCollection, sourceId, referenceDateField)`: it creates rows for
timings that aren't there, deletes ones that aren't desired, and leaves
matches alone. Call it again whenever the user changes preferences.

### Custom sync logic

If a single source record has multiple date fields (the people module has
both `birthday` and `anniversary`), the per-field reconciliation is
slightly more involved. Mirror the pattern in
[`frontend/src/modules/people/utils/notificationSync.ts`](../frontend/src/modules/people/utils/notificationSync.ts):
one helper per date field, called in parallel from create/update mutations.

Don't forget the cleanup path — when the source record is deleted, delete its
recurring notifications too. See `deleteRecurringNotificationsForPerson` in
the same file.

### Read-side hooks

```ts
// All recurring rows for a single source record (e.g. one person)
const { data } = useRecurringNotifications('people', personId);

// Every recurring row for the current user (settings/admin views)
const { data } = useAllRecurringNotifications();
```

Both come from `@/modules/notifications`. Filtering happens client-side
because aepbase doesn't accept arbitrary field filters; lists return all
of the user's rows under the parent path.

## Delivery path 2: immediate notifications

Use this when an action should fire a push right now (a chore was assigned,
a recipe was updated, a one-off reminder).

### From a server route

Wrap your handler with `sendUserNotification`:

```ts
// frontend/src/app/api/<feature>/notify/route.ts
import { NextRequest } from 'next/server';
import { sendUserNotification } from '../../notifications/utils/send-user-notification';

export async function POST(request: NextRequest) {
  return sendUserNotification(request, {
    title: 'Chore assigned',
    body: 'You picked up "Take out trash"',
    tag: 'chore-assigned',
    url: '/chores',
    sourceCollection: 'chores',
    sourceId: '<chore-id>', // optional
  });
}
```

`sendUserNotification` ([`utils/send-user-notification.ts`](../frontend/src/app/api/notifications/utils/send-user-notification.ts))
handles VAPID, fetches the caller's `notification-subscriptions`, sends the
push, prunes expired endpoints, and writes the `notifications` row. It
authenticates from the request's bearer token, so the notification always
goes to the **calling** user.

`/api/notifications/send-test` and `/api/notifications/send-grocery` are
two-line wrappers around this helper — copy them when you need a new
fire-and-forget endpoint.

### From the cron, on behalf of another user

To notify a user other than the caller (e.g., an admin/cron triggers a push
to user X), use `sendImmediateNotification` from
`utils/notification-sender.ts`. It takes an admin token and a `userId`:

```ts
await sendImmediateNotification(adminToken, {
  userId: targetUserId,
  title: '...',
  message: '...',
  sourceCollection: 'recipes',
  sourceId: recipeId,
});
```

You're responsible for obtaining the admin token (typically by reusing
`loginAdmin()` from the cron path). Don't expose this from a client-callable
route without an authorization check.

## Reading notifications in the UI

The notifications module ships these hooks (importable from
`@/modules/notifications`):

| Hook                              | Returns                                       |
|-----------------------------------|-----------------------------------------------|
| `useNotifications()`              | All `notifications` rows for the current user |
| `useNotificationStats()`          | `{ total, unread, read }`                     |
| `useMarkNotificationAsRead()`     | Mutation, takes a notification id             |

Plus `useUnreadNotifications()` from the dashboard module for a top-N inbox
preview.

For your own module, you usually don't need to render notifications — link
to `/notifications` (the unified inbox at
`frontend/src/app/(app)/notifications/page.tsx`). If you do want a
module-scoped feed, query `notifications` filtered client-side by
`source_collection === '<your-collection>'`.

## Adding notifications to a new module

Checklist for a module that owns records with reminder dates:

1. **Pick a `reference_date_field`** on your aepbase resource. It must be a
   stored ISO date/datetime string on the source record.
2. **Surface the timing UI.** Reuse `NOTIFICATION_TIMING_OPTIONS` from
   `@/modules/notifications` to render checkboxes/toggles. Store the user's
   selection on the form state, not on the source record (it's encoded in
   the recurring rows).
3. **Sync on create/update.** In your create/update mutation, after the
   record is saved, call `useSyncRecurringNotifications` (or write a
   per-field helper like `notificationSync.ts` if there are multiple date
   fields). Wrap it in try/catch and log — a sync failure shouldn't fail
   the user's primary save.
4. **Clean up on delete.** In your delete mutation, list recurring rows for
   `(your_collection, source_id)` and remove them. Aepbase doesn't cascade.
5. **Invalidate caches.** After mutations, invalidate
   `queryKeys.module('recurring_notifications').list()` so the global
   recurring view stays consistent.
6. **Pick a `source_collection` value that matches your aepbase plural.**
   The cron does `aepGet(source_collection, source_id, ...)`, so the value
   must be a real collection name — e.g. `'people'`, not `'person'`.
7. **Make sure your source record has a `name` field** if you want
   `{{name}}` to resolve in templates. If not, hardcode the title or extend
   `processTemplate` in `notification-sender.ts`.

For immediate notifications:

1. Add a route under `frontend/src/app/api/<feature>/...` that calls
   `sendUserNotification`.
2. Trigger it from the client with a regular `fetch` (forward the user's
   aepbase bearer token via the standard `aepbase` wrapper, which the
   server `authenticate` helper validates).
3. Decide on a stable `tag` so repeated pushes collapse rather than stack.

## Testing

- **Unit:** mock `aepbase` (already done globally in `src/test/setup.ts`)
  and assert your sync helper creates / deletes the right recurring rows
  for given inputs. The people module's tests are a good reference.
- **End-to-end:** the cron is hard to drive from Playwright. For e2e,
  prefer asserting that recurring rows land in aepbase after a UI
  interaction (seed via REST helpers, then use the UI). Don't try to
  observe a real push delivery in CI.
- **Manual smoke test:** with the dev stack running, `POST` to
  `/api/notifications/send-test` from the browser DevTools console (the
  Settings page wires this up via `useSendTestNotification`). You should
  see a push and a new row in your inbox.
- **Cron locally:** `curl http://localhost:3000/api/notifications/cron` —
  no `CRON_SECRET` is required when the env var is unset. Make sure
  `AEPBASE_ADMIN_EMAIL`/`AEPBASE_ADMIN_PASSWORD` and the VAPID keys are in
  `frontend/.env.local`.
